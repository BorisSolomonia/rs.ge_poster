## 4. Findings & fixes (5-step deep dives)

### B4 — ConfigStore O(n) scans + per-call regex compile — FIXED (commit `ccbf659`)
1. **Symptom:** matrix p95 927 ms @5VU; JFR shows ~40% of CPU in regex machinery; reconciliation scales O(suppliers × products).
2. **Trigger:** any cash-flow request over a large range (2 lookups/transaction × 50k) and any reconciliation run (per-supplier product-mapping filter over the full product list).
3. **Code flaw:** `ConfigStore.normalizeSalesKey` ended in `.replaceAll("\\s+", " ")` — `Pattern.compile` on every call; `findById`/`findSupplierMappingByPosterAlias`/`getProductMappings`/`isSalesProductExcluded`… were `stream().filter().findFirst()` linear scans invoked inside hot loops (`CashFlowService.ruleCandidates:485,489`, `ReconciliationEngine:78-79`, `SalesAnalysisService:63`).
4. **Architectural gap:** ConfigStore was designed as list-of-records + full scans; adequate at 14 seed rows, quadratic once callers loop over suppliers/transactions. No derived-index layer; regex not hoisted despite an existing precompiled-pattern idiom (`TAX_ID_PATTERN`).
5. **Root correction:** six derived HashMap indexes rebuilt wholesale from the authoritative lists in every `persist*()`/`load()` under the existing RW-lock (rebuild-not-maintain ⇒ no drift path); `static final Pattern WHITESPACE`. Tie-breaking preserved via `putIfAbsent` in list order. **Gate:** 94/94 tests, 7/7 goldens identical, S1 p95 927→757 ms (−18%), S2 876→678 ms (−23%), throughput +23%.

### B1 — cash-flow per-request recompute (resolution, fingerprints, rules, result) — FIXED (commit `16ca7a6`)
1. **Symptom:** even after B4, matrix/forecast p95 ~750/680 ms; JFR: SHA-256+hex per transaction per request, `findRule` linear scan per candidate.
2. **Trigger:** every matrix/forecast/transactions request re-resolves all 50k transactions (3 candidates × 45 rules each) and re-hashes fingerprints.
3. **Code flaw:** `CashFlowService.resolve`/`findRule` (:496-512) + `CashFlowFingerprint` (SHA-256/`normalizeTin` with per-call `replaceAll("[^\\d]","")`) executed inside the per-request loop; matrix DTO rebuilt every request.
4. **Architectural gap:** no versioning concept for rules/overrides/categories ⇒ nothing could be cached safely; results treated as always-stale.
5. **Root correction:** monotonic version stamp incremented on every rule/override/category mutation (service layer is the sole writer — future direct store writers must bump it, noted in commit); Caffeine caches for per-transaction Resolution+fingerprint (200k) and whole matrix/forecast DTOs keyed (range|asOf, version) with 60 s TTL matching the pre-existing sourceCache staleness window; rule index Map<(matchType,value)> preserving direction-specific > agnostic precedence; precompiled TIN pattern. **Gate:** 99/99 tests (4 new: cache-hit identity, 2× version-bump invalidation, precedence lock), goldens byte-identical, matrix p95 927→**2 ms**, forecast 876→**1 ms**, cache-hit 10 ms single-request, cold key ~755 ms (recompute now happens only when data or rules actually change).

### B2 — sales-analysis O(products × buckets × rows) — FIXED (commit `a22a574`)
1. **Symptom:** S3 upload p50 2.4 s, p95 3.2 s, max 9.8 s @2VU.
2. **Trigger:** every sales-analysis upload re-filters each product's rows for every time bucket; day-cursor loops copy + rescan maps per bucket.
3. **Code flaw:** `buildProductSeries` (:265-283) nested option×bucket×rows filtering; `sumRange` did `new TreeMap<>(source)` full copy per call.
4. **Architectural gap:** bucketing computed from scratch per (product,bucket) pair instead of one partition pass; interval structure of buckets (contiguous, non-overlapping) never exploited.
5. **Root correction:** single pass assigning each row to its owning bucket via TreeMap.floorEntry on bucket starts; per-product BigDecimal accumulators; NavigableMap.subMap slices for cursor loops. Output equivalence proven (exact BigDecimal addition + partition ⇒ identical pre-round sums; MoneyUtil HALF_UP scale-2 at emission) and locked with a new multi-product interleaved-dates regression test. **Gate:** goldens byte-identical, upload p50 2 400→**1 664 ms** (−31%), p95 3 207→**2 612 ms** (−19%), throughput +71%. Remaining cost is Apache POI workbook parsing + the 18.8 MB response (see B9) — the compute side is fixed.

### F1 — CashFlowPage: zero memoization, charts re-render on row expand — FIXED (commit `426266a`)
1. **Symptom:** probe: expanding 3 matrix rows re-rendered the charts subtree 4×/185 ms (chart data unchanged).
2. **Trigger:** any expand/collapse or local state toggle on the page.
3. **Code flaw:** recursive `MatrixCategoryRow` not memoized; per-cell inline closures; `CashFlowCharts` re-aggregates (flatMap/sort/slice + per-month reduce) on every parent render with inline recharts prop objects.
4. **Architectural gap:** page-level state and derived chart data coupled into one render tree with no stability boundaries.
5. **Root correction:** React.memo on rows/charts with stable useCallback handlers and useMemo'd chart aggregation. Subtlety worth knowing: a naive per-row `expanded` boolean memo is WRONG for a recursive tree — a bailed open ancestor never reconciles its children, freezing nested toggles. The fix uses a custom comparator that skips the unstable `openCategories` Set ref but forces re-render of open-with-children rows, so only the toggled path re-renders. **Gate:** probe shows charts **0 ms** on expand (was 4×/185 ms); tsc + eslint + build green.

### F2 — BudgetForecast: whole-table re-render per cell edit — FIXED (commit `4fd0823`)
1. **Symptom:** probe: editing 2 cells → forecast-chart 49×, rows 36×, table 12× re-renders.
2. **Trigger:** every keystroke/commit in the editable matrix.
3. **Code flaw:** per-render `new Map(...)` construction (:100, :210), inline per-cell `onCommit` closures, no memo on rows/cells/chart.
4. **Architectural gap:** editing state and derived render data live in the same unstable render pass.
5. **Root correction:** useMemo'd totalsByKey/cellsByKey, memoized rows/cells with row/period identity pushed into the cell (killing per-cell closures), memoized chart data. **Gate:** probe shows forecast-chart **0 renders** on cell edit (was 49×/145 ms), rows 36→19, table 12→7; tsc + eslint + build green.

### F3 — no build tuning; vendor monolith — FIXED (commit `e3e0b02`)
1. **Symptom:** 220 KB vendor chunk invalidated by any app change; recharts split across 2 chunks.
2. **Trigger:** every deploy (cache churn), every chart-page visit (2 fetches).
3. **Code flaw:** `vite.config.mjs` had no `build` block at all.
4. **Architectural gap:** build config never revisited after scaffold.
5. **Root correction:** `manualChunks` — react 164 KB / query 78.5 KB / recharts 406 KB (consolidated) / app index 26.5 KB. Framework chunks now cache-stable across deploys. Honest note: total raw bytes ≈ unchanged; the win is cache granularity + one recharts fetch. recharts itself (with decimal.js-light + fast-equals) remains ~60% of chart-page JS — replacement evaluation parked (§6).

## 5. Additional findings (discovered while measuring)

- **B8 (correctness, not perf):** Poster supplies parser is configured `skip-header-rows: 2` but `export_supplies_260224 (2).xlsx` has 1 header row → **first data row (doc 1610) silently skipped** in reconciliation. Recommend: per-file header detection or config audit. (Found while spec'ing upload requests against parser code.)
- **B9:** `POST /sales-analysis/analyze` returns an **18.8 MB JSON** response for a modest workbook. No pagination/streaming; verify gzip is enabled at the proxy (Caddy) — otherwise this dominates real-world latency far beyond compute. Recommend: response slimming (omit per-row echoes; fetch series on demand) or `Content-Encoding` verification in prod.
- **Session/permission note:** the app has no authentication layer at all (no spring-security, no filters, API key absent); acceptable behind the proxy only if the proxy enforces auth — flagged for verification, out of audit scope.

## 6. Parked items (root cause analyzed, correction designed, not executed)

### B3 — blocking bank/SOAP I/O on ForkJoinPool.commonPool
1. **Symptom (latent):** under concurrent supplier-debt refresh + any parallel-stream work, requests can stall up to the 120 s bank timeouts.
2. **Trigger:** `SupplierDebtService` fan-out (`:202,205,208,459-467`) + `RsgeSoapClient` chunk fan-out (`:170`) submit blocking HTTP/SOAP calls to the shared commonPool (sized ~cores−1; on the 0.5-CPU prod container ≈ 1 thread).
3. **Code flaw:** `CompletableFuture.supplyAsync(...)` without an executor argument, everywhere.
4. **Architectural gap:** no owned, bounded, named executor for external I/O; latency domains (bank I/O vs CPU work) share one pool.
5. **Designed correction:** `@Bean("bankIoExecutor") ThreadPoolTaskExecutor` (core 4/max 8, bounded queue, named threads, 130 s keep-alive) passed to every supplyAsync; verify via slow-stub unit test asserting commonPool untouched (`Thread.currentThread().getName()` prefix) + existing SupplierDebtServiceTest. Parked at T+70 per time-box; low regression risk, ~1 h with tests.

### B5 — unused Google Sheets/Drive/auth dependencies in pom.xml
Zero references in `src/` (verified by grep). Removal shrinks the jar and CVE surface. Parked: needs one clean `mvn verify` + jar-size before/after — mechanical.

### F4 — QueryClient/polling tuning
`staleTime` 30 s, Dashboard polls every 30 s, no `gcTime` tuning, `refetchOnWindowFocus` default-on. Tuning changes data-freshness semantics — product decision needed (e.g., `refetchOnWindowFocus: false` + `gcTime: 10 min` for analysis pages). Parked deliberately, not for time.

### B6 — SourceLedgerStore unbounded on-disk growth
Ledger JSON grows with every synced statement forever; only RAM caches are capped. Designed correction: retention window (e.g. 36 months) + compaction on `mergeWindow`, or monthly ledger file partitioning with lazy month loading. Needs a data-retention decision from the business first.

### EOL policy
The tree accumulated CRLF↔LF churn (67 files, whitespace-only — committed as `0ddbc23` after `git diff -w --stat` verification). Recommend committing a `.gitattributes` (`* text=auto eol=lf` + explicit binaries) in a quiet week to end the churn class permanently.

### recharts weight
recharts + transitive deps ≈ 60% of chart-page JS (406 KB raw / 110 KB gz). If chart needs stay at line/bar/area, a lighter lib or direct d3 modules could halve chart-page weight. Product/design decision; parked.

## 7. Measurement integrity notes

- The React render probe writes REAL forecast overrides through the dev proxy (it edits actual cells). This polluted one baseline golden (`overridden: true`) and was caught by the golden gate itself; the override was deleted via the API and the protocol amended: **goldens before probes; override cleanup after every probe run.** Probe instrumentation (`PERF-PROBE`-marked `<Profiler>` wrappers + `data-probe` attributes + `src/perfProbe.ts`) is stripped before commits and never ships.
- Windows locks a running jar: stop the backend before `mvn verify` or repackage fails with a rename error (this cost one diagnosis cycle; documented for CI-on-Windows too).
- All numbers are relative comparisons on one warm machine with pinned JVM resources; absolute production numbers will differ, deltas should hold.
