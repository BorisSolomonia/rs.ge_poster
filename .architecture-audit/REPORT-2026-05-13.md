**Date:** 2026-05-13
**Auditor:** `/architecture-intelligence-auditor`
**Scope:** Audit + refactor proposal (Phases 1–7 + 10). No edits applied.
**Profile:** Small team, internal-use; benchmarks weighted toward modular-monolith pragmatism.
**Branch:** `main` @ `940e3f5 "TBC_VM"`

---

## 1. Executive Summary

Camora ERP is a single-operator internal reconciliation/ERP tool — a Spring Boot 3.2 / Java 21 backend plus React 18 / Vite frontend, deployed as two containers to one GCP VM behind a shared Caddy proxy under `/camora`. It integrates four external sources (RS.ge SOAP, Poster XLSX exports, Google Sheets ledger, TBC DBI + BOG banking) and persists business state as JSON files on disk. There is no database, no authentication, no HTTPS at the proxy, and the test suite is three files.

**The architecture choices are mostly defensible for a one-VM internal tool.** Modular monolith with file persistence + Docker Compose is appropriate for the operator's scale. The team has already authored two thoughtful docs (`SECRET_AUDIT_REPORT.md`, `PRODUCTION_HARDENING_CHECKLIST.md`) that name nearly every blocker for going public. The audit confirms those blockers and adds a smaller layer of concrete code-level findings.

**The single dominant risk is not technical — it is operational.** Real bank/RS.ge credentials are committed to git history (`backend/.env`, `frontend/.env`, `docker/.env.example`), flagged by the team on 2026-05-08, but **the rotation+history-rewrite cleanup was never executed**. As of 2026-05-13 those values are still readable on `main`. Until that is fixed, no architectural improvement matters: the system is one `git clone` away from compromise.

**Headline severity counts** (file:line cited in §10/§11):
- Critical (P0): 2 — leaked credentials still on `main`; certificate `.pfx` path hardcoded to a local Windows path.
- High (P1): 6 — no auth on bank-mutating endpoint, no HTTPS, persistence not migrated to DB, full-runtime DEBUG default, unbounded ForkJoinPool fan-out to RS.ge, multipart limit too low for real exports.
- Medium (P2): 9 — duplicated `levenshtein`/`normalizeSalesKey`, 1,203-line `CamoraProperties` god-config, no rate-limiting/pagination on heavy endpoints, no readiness checks for required integrations, test coverage <10%, Caddy `respond /health 200` masks proxy outages, etc.
- Low (P3): 12 — see §10/§11.

**Top recommendation:** before any refactor — execute the rotation drill the team already wrote. Then ship the small auth shim. Then tackle persistence migration. Code-level refactors (§7) are second-order until those three are done.

## 2. Application Purpose

Camora ERP serves a single Georgian SMB operator (Verapani / Camora) who needs to reconcile what their POS (Poster) recorded against what the Georgian tax service (rs.ge) recorded, classify cash-flow movements from a Google Sheets ledger, ingest live bank statements from TBC DBI and BOG Business Online, and track supplier debt. All inputs land in one of three forms: a multipart XLSX/CSV upload, a SOAP call, or a scheduled Google Sheets pull. All outputs are tabular dashboards rendered by React. There is one user, one VM, and a sister `orderapp` deployment that shares the proxy and the GCP Artifact Registry repo.

**Observations:**
- **No circular deps** between Java packages.
- **`ConfigStore` is a hub.** Six services + several controllers depend on it. This is the application's single source of truth, by design.
- **Hidden coupling**: `GlobalExceptionHandler` imports `TbcDbiClient` only to read its `public static final String` codes (`TbcDbiClient.PASSWORD_CHANGE_REQUIRED`, etc.). This couples a config-layer class to a feature module. Low severity but flagable.
- **Helper duplication**: `levenshtein` is implemented twice — once in `ConfigStore.java:886` and once in `CashFlowService.java:1014`. `normalizeSalesKey`/`normalizeCategory` is called from six classes (good — centralized in `ConfigStore`).
- **Frontend ↔ backend coupling**: the `axios` modules use string URLs that mirror the `${camora.api-prefix}` Spring routes. No code-gen, no shared types. Drift risk on rename.

## 6. Module Interaction Analysis

Two representative request flows traced end-to-end:

**Flow A — `POST /api/v1/reconciliation/analyze`** (file:line):
1. `ReconciliationController.analyze` (`ReconciliationController.java:46`) receives multipart `rsgeFile`, `posterFile`, `dateFrom`, `dateTo`.
2. Validates `dateTo >= dateFrom` (`ReconciliationController.java:110`).
3. Parses RS.ge CSV via `RsgeCsvParser.parse(InputStream)` and Poster XLSX via `PosterXlsxParser.parse(InputStream)` — both read columns by index from `CamoraProperties.Parsers`.
4. Calls `ReconciliationEngine.run` (`ReconciliationEngine.java:33`). The engine reads all supplier mappings out of `ConfigStore`, groups records by supplier-raw key, computes per-supplier totals, marks discrepancies with `MoneyUtil.isMatch`, and writes any newly discovered supplier into `ConfigStore.registerStandaloneSupplier`.
5. Result is stored in `ResultCache` (in-memory; 12-hour TTL).
6. Response is `ApiResponse.ok(result)`.

**Flow B — `GET /api/v1/cash-flow/overview`** (file:line):
1. `CashFlowController.overview` (`CashFlowController.java:39`) invokes `CashFlowService.getOverview(from, to)`.
2. On first call the service kicks `refreshSnapshot()` (`CashFlowService.java:109`), which calls `GoogleSheetsCashFlowClient.fetchLedgerRows()` — a service-account-authenticated read against the configured Google Sheet ID.
3. Rows are parsed by column index (`CamoraProperties.CashFlow.columns`), classified into INCOME/EXPENSE/SAFE/DIVIDEND/UNCATEGORIZED by keyword match against config (`CashFlowService.java:578`).
4. Snapshot is stored as `volatile CashFlowSnapshot` and refreshed by `@Scheduled` every 4h (`CashFlowService.java:98`).
5. Response is filtered by date range, aggregated by month, returned as `CashFlowOverviewDto`.

**Findings:**
- Bank-fetch endpoints (`GET /api/v1/bank-analysis/tbc`, `GET .../bog`) trigger synchronous SOAP/REST against external systems with up to 120-second timeout. There is no async/queueing. A slow upstream blocks a Tomcat thread.
- Reconciliation result discovery side-effects (writing new suppliers) happen *during* a read-style HTTP POST. If the request fails midway after writes, the file state diverges from the response (`ReconciliationEngine.java:176`).
- `CashFlowService.refreshSnapshot()` swallows the previous overview on failure (`CashFlowService.java:131`) but keeps the old `rows`, so stale data is served indefinitely until success. There is no max-stale boundary.

## 7. Benchmark Architecture Comparison

The application is best characterized as an **ERP / internal back-office SaaS** with **modular monolith + external SOAP/REST integrations** topology. Three reference architectures relevant for that category and the small-team profile:

### Benchmark 1 — Single-VM Modular Monolith + File Store (e.g. early-stage SaaS, ERP for one company)
- **Structure:** one process, packaged modules, file-system or single-node DB.
- **Strengths:** lowest operational cost; single deployable; one team can change cross-cutting concerns atomically.
- **Weaknesses:** no horizontal scaling without rework; deploys touch the whole app; persistence reliability depends on backups.
- **Scalability:** vertical only.
- **Maintenance cost:** lowest of the three.
- **Adaptability:** high while feature surface is small; degrades quickly past ~30 endpoints + multiple integrations.

### Benchmark 2 — Modular Monolith + RDBMS + Background Workers
- **Structure:** Spring Boot monolith with JPA/Flyway, a Postgres in Docker Compose, `@Scheduled` or Quartz for batch jobs, async controllers for slow external calls.
- **Strengths:** transactional integrity, audit trail naturally lives in DB tables, horizontal-ready (sticky-session-free with externalized state), well-documented Spring patterns.
- **Weaknesses:** must manage DB lifecycle (migrations, backups, version bumps).
- **Scalability:** medium — can scale stateless backend N-wide once state is in Postgres.
- **Maintenance cost:** medium.
- **Adaptability:** high; most ERP-shaped apps end up here.

### Benchmark 3 — Hexagonal / Clean Architecture with Adapters
- **Structure:** domain core in pure Java, adapter modules for SOAP/REST/Google Sheets/JPA, controllers as input adapters.
- **Strengths:** swap-in new bank integration without touching reconciliation; high testability (pure-domain unit tests).
- **Weaknesses:** more files, more interfaces, more ceremony — overkill for a single-operator app.
- **Scalability:** same as B2.
- **Maintenance cost:** high (until team grows).
- **Adaptability:** highest; but only worth it past ~5 dev team.

## 9. Technical Debt Analysis

Concrete, file:line-cited debt items:

1. **`ConfigStore.java:886-905`** — `levenshtein` method duplicated in `CashFlowService.java:1014-1035`. Identical implementations.
2. **`CamoraProperties.java`** — 1,203 lines of getters/setters; would split naturally into `IntegrationProperties` (rsge/tbc/bog), `ParsingProperties`, `CashFlowProperties`, etc.
3. **`CashFlowService.java`** — 1,070 lines, mixes parsing, classification, reporting, debugging logs, and analysis. Three clear responsibilities collapsed into one service.
4. **`ConfigStore.java`** — 905 lines, eight different domain mutations behind one lock. Each `upsert*WithRollback` reimplements identical try/catch logic.
5. **`GlobalExceptionHandler.java:48-57`** — imports `TbcDbiClient` solely to read static-final code constants. Move codes to `TbcDbiException` or a dedicated `BankErrorCodes`.
6. **`RsgeSoapClient.java:117-136`** — unbounded ForkJoinPool fan-out.
7. **`TbcDbiClient.java:60-78`** — `while(true)` paging loop with `MAX_PAGES_PER_REQUEST=1000` (~700 k rows). Acceptable but should have a date-range cap upstream.
8. **`ReconciliationEngine.java:176-177`** — side-effecting writes inside an analyze flow. Should be opt-in.
9. **`CashFlowService.java:131-140`** — silently keeps stale rows on refresh failure with no max-stale boundary.
10. **`backend/src/main/resources/application.yml:178-181`** — `CAMORA_LOG_LEVEL` defaults to `DEBUG` for `ge.camora.erp`. Combined with `RSGE_DEBUG=false` (line 31), credentials don't leak today, but a future flip is one env-var away.
11. **No test coverage** for reconciliation, cash flow, bank analysis services, parsers, or SOAP clients. The only tests are `SupplierDebtServiceTest`, `TbcDbiClientTest`, `BogBusinessOnlineClientTest`.
12. **No `@Transactional`** anywhere because there's no DB; but several `ConfigStore` methods write multiple files (e.g. `deleteSupplierMapping` writes both `supplierMappings` and `productMappings`). If the second write fails, in-memory state is inconsistent with disk.
13. **`docker/compose.production.yml`** — backend memory limit `768M` with `MaxRAMPercentage=75` → 576 MB heap. POI XLSX parsing of 50 MB files (the multipart limit) can blow heap. No swap reservation.
14. **CORS allows `*` headers** (`application.yml:29`) — combined with `allowCredentials=false` it's safe today, but the prod `<VM_PUBLIC_IP>` placeholder pattern means the deployed CORS may not actually match the deployed host name.
15. **Frontend ships no tests** (`AGENTS.md:40` acknowledges).
16. **`docker/.env.example` is not an example** — it contains real values (see §11).

## 16. Remaining Risks

After this audit, the system still carries the following unaddressed risks (in priority order):

1. **Secret rotation has not happened.** Until executed, every clone is a compromise.
2. **No authentication.** Any device on the VM's network (or the public internet, depending on firewall posture) can call `POST /api/v1/bank-analysis/tbc/password-change` and lock out the operator.
3. **No HTTPS at the proxy.** Cleartext traffic includes bank query parameters and (post-auth) session cookies.
4. **No database.** Disk corruption on the VM volume `camora_data` is unrecoverable without backups. The compose file does not define a backup strategy.
5. **Synchronous external calls** can block the request thread for up to 120 s × N pages.
6. **One-VM single-point-of-failure.** Acceptable today, blocking later.

## 18. Adaptability Analysis

**Where new features land cleanly:**
- New bank integration → add a `Client` + `Exception` under `module.bankanalysis.*`, wire into `BankAnalysisService.analyze`. Pattern is symmetric for TBC/BOG.
- New mapping type → add an entity in `model.config.*`, a config-files entry in `CamoraProperties.ConfigFiles`, a list field + persist method in `ConfigStore`, a controller. Pattern is already followed 8 times.
- New cash-flow group/keyword set → entirely config (`CAMORA_CASH_FLOW_*_KEYWORDS`).

**Where new features fight the architecture:**
- Multi-tenant or multi-company → JSON store is one-company-flat; would require introducing a tenant key into every entity. Cleanly a P3-level rewrite.
- Real-time updates → no WebSocket / SSE infrastructure; would require new endpoint stack.
- Mobile companion app → no auth, so it can't safely connect from outside the VM today.
- Reconciliation history / audit → no persistence of historical runs (in-memory Caffeine for 12 h).

## 19. Scalability Analysis

| Dimension | Today | Limit | Bottleneck |
|---|---|---|---|
| Concurrent users | 1 (designed) | ~10 before file-lock contention shows up | `ConfigStore.lock` serializes all writes |
| Reconciliation runs / hour | ~10 with 50 MB XLSX | ~30 before POI heap pressure on 768 MB container | JVM heap = 75% of 768 M = 576 M; POI XSSF parses ~150 MB raw |
| Cash-flow refresh | every 4h | unchanged | Google Sheets quota: 60 reads/min — irrelevant at this rate |
| Bank statement pulls | one date range per call | 700-row pages × 1000 = 700 k rows max per request | `MAX_PAGES_PER_REQUEST=1000` in `TbcDbiClient` |
| RS.ge fetches | parallel across 3-day chunks | unbounded — **DoS risk** | `RsgeSoapClient.fetchInChunks` (B1) |
| HTTP throughput | Tomcat default 200 threads | hits ~50 with synchronous 120s SOAP timeouts | thread starvation |
| Storage | 8 JSON files, KB-sized | MBs each before performance degrades | `objectMapper.writerWithDefaultPrettyPrinter` writes whole file every change |
| Horizontal | single instance only | cannot scale | shared writable volume + file lock |

**Headroom verdict:** the current setup comfortably serves 1–3 internal users. To serve 10+ users, the `ConfigStore` → Postgres migration (P3-1) and the cap on RS.ge fan-out (P1-3) are the unavoidable changes. Most other concerns are policy/operational, not throughput.

## 20. Long-Term Architecture Recommendations

If Camora ERP stays internal (one company, ≤5 users):
- Stay on the modular monolith. Resist microservice gravity.
- Do P0 + all of P1 in the next 2–4 weeks.
- Do P2 over the following 2–3 months as time allows.
- Do P3-1 (Postgres) the moment you start adding compliance/audit-trail needs or onboard a second user with mutation rights. Defer indefinitely otherwise.

If Camora ERP grows to a multi-tenant SaaS (multiple companies):
- Postgres + tenancy column is non-negotiable. Day 1 of the SaaS pivot.
- Adopt Spring Modulith (or just enforced package-private with a CI check) to keep module boundaries.
- Move integrations (rs.ge, TBC, BOG) behind explicit adapter interfaces in a `port/` package so swapping providers per-tenant is feasible.
- Replace `volatile CashFlowSnapshot` with a per-tenant cache (Caffeine keyed by tenant + range).
- Bring in a real BFF or OpenAPI codegen to keep frontend/backend in sync.
- Keep the Caddy shared-proxy pattern — it survives the pivot.
